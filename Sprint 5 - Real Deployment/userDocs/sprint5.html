Real Components

Real OutSonar

By requirements, the outsonar placed into the outdoor area must be a real one. The real outsonar is deployed on a Raspberry Pi Board.
To misure distances we use a HC-SR04 distance sensor. Note that the customer provided us the code to control this sensor by means of a C program. However, we decided to use Python to control the real components of our system, so the code to control the HC-SR04 can be found here. In order to integrate the real sensor in our system we also implemented the RealOutSonarSupport.kt which is the object used by the outsonar QActor (only when the configuration is "real") in the way illustrated in Sprint 3.

Since the OutSonar configuration could be both real or virtual, we also added the possibility to ask the system in which configuration the outsonar is working. So, when an outsonar_config request is sent, it is replied by means of outsonar_real or outsonar_virtual response.
This information is useful to the MockDataGUI in order to show the outsonar distance control slider only when the configuration is "virtual".

Real Fan

By requirements, the fan is not required to be real. However, since we designed the system to work with both a virtual fan and a real one, and since we obtained such component at no cost, we decided to deploy the system with this additional real component on a Raspberry Pi Board.
The real fan is just an on/off component and only a raspberry GPIO pin is required to control it.
The code (written in Python) that permits to switch the fan on/off can be found here.
In order to integrate the real fan in our system we also implemented the RealFanSupport.kt which is the object used by the fan QActor (only when the configuration is "real") in the way illustrated in Sprint 4.

Distributed System

Distributed System

Each component is deployed in a Docker container. All the containers are publicly available in our Docker Hub.
For each container there is a different docker-compose file.yaml that allows each component of the distributed system to be started on the desired machine.
In addition, we also provide a bash script to start/stop the system in an automatic way. They are mainly intended to be used for the demo we will show to the customer.
These .yaml files and the start/stop script can be found in the run directory.

DNS

We use a local DNS service to simplify the relocation of the distributed system components.
Thanks to the hostname-ip mapping service, offered by the DNS, we may in the future change the ip addresss of a device (in which a component resides) without change the configuration of the entire system.

Requirements Text

2) Ipotizzare che la temperatura sia e permanga ok, il fan spento, INDOOR e OUTDOOR libere.
3) Simulare che il sistema riceva, a distanza di 2 sec l'una dall'altra:

     b) una carenter

4) Definire le variazioni di stato che il SISTEMA REALIZZATO effettua.
   Ad esempio:
   Il sistema risponde al client al punto a) e poi inizia una acceptin al punto b).
   In questa fase, il cliente vede ... mentre il manager vede ...
   Quando arriva c) ...
5) Indicare la strategia seguita per poter automatizzare un test di questo tipo.

Test Preconditions

To meet the test preconditions (see the first test requirement), the system must be started with a properly configured parking slot database: i.e. one parking slot occupied and the other ones free.

System behavior

Request name in test description Request name in ParkServiceGUI POST API Request name in domain logic (QAK)
enterrequest /parking_car_interest parking_car_interest
carenter /car_enter car_enter
pickup request /car_pick_up car_pick_up
  1. The system receives a enterrequest and reserve a parking slot (SLOTNUM_A). The reserved parking slot (SLOTNUM_A) is sent as reply to the client.
    The client sees the SLOTNUM_A.
    The manager sees a status transition (for both INDOOR AREA and SLOTNUM_A) from FREE to RESERVED.
  2. The system starts to observe the weightsensor.
    The manager sees an INDOOR AREA status transition from RESERVED to OCCUPIED when a car in the INDOOR AREA is detected.

  1. The system receives carenter and starts the transport trolley car park task.
    The manager sees a transport trolley status transition from IDLE to WORKING.
  2. The transport trolley is moved from HOME to INDOOR.
    The manager sees the transport trolley in movement.
  3. The transport trolley takes the car from the INDOOR AREA and the system reply to carenter with the TOKENID_B. The system stops to observe the weightsensor.
    The client sees the TOKENID_B.
    The manager sees a status transition from OCCUPIED to FREE in the INDOOR AREA.
  4. The transport trolley is moved from INDOOR to SLOTNUM_A.
    The manager sees the transport trolley in movement.
  5. The transport trolley release the car at SLOTNUM_A and the persistent status of the parking slot (SLOTNUM_A) becomes OCCUPIED.
    The manager sees a SLOTNUM_A status transition from RESERVED to OCCUPIED.
    Note: a transport trolley back to home task might be initiated if the task of step c) has not yet been started.

  1. The system receives a pickup request and validate the TOKENID_C (request input).
    The manager sees an OUTDOOR AREA status transition from FREE to RESERVED.
  2. The TOKENID_C is valid (mapped to parking slot SLOTNUM_C) so the system starts the transport trolley car pick up task.
    Note: a transport trolley status transition from IDLE to WORKING might happen if the transport trolley was not already in WORKING state.
  3. The car pick up task is delayed until the car park task of step b) is running...
  4. The transport trolley is moved from its actual position (time-dependent, probably the SLOTNUM position of step b)) to SLOTNUM_C position.
    The manager sees the transport trolley in movement.
  5. The transport trolley takes the car from SLOTNUM_C and the persistent status of the parking slot (SLOTNUM_C) becomes FREE.
    The manager sees a SLOTNUM_C status transition from OCCUPIED to FREE.
  6. The transport trolley is moved from SLOTNUM_C to OUTDOOR.
    The manager sees the transport trolley in movement.
  7. The transport trolley release the car in the OUTDOOR AREA and the system starts to observe the outsonar.
    The manager sees an OUTDOOR AREA status transition from FREE to RESERVED.
    The manager sees an OUTDOOR AREA status transition from RESERVED to OCCUPIED when a car in the OUTDOOR AREA is detected.
  8. The system reply with success to the pickup request.
    The client sees that its car is now in the OUTDOOR AREA.
  9. The client takes the car from the OUTDOOR AREA. The system stops to observe the OUTDOOR AREA.
    The manager sees an OUTDOOR AREA status transition from OCCUPIED to FREE.
  10. Since there are not other request, the transport trolley start the back to home task.
    The manager sees the transport trolley in movement.
    The manager sees a transport trolley status transition from WORKING to IDLE when the transport trolley is arrived at HOME.
Note: c.9) and c.10) might run concurrently.
Note: timer(s) like DTFREE and DTOCCUPIED are not discussed in this description since they have not effect in this test that simulate a correct client and system behaviour.

In order to automate the test, we have written a Kotlin file which makes use of the JUnit framework.
To interact with the system, the test uses the POST APIs exposed by our webservers.
Moreover, to retrieve the current status of the system, the test register itself to the STOMP service
offered by the Park Service Status GUI.
Note that by doing this, the test simulates the behavior of the client when he interacts with the GUI's buttons.

Deployment

We used Docker to deploy the software developed during this sprint.
Below there is a guide about how we start the system in automatic way in order to show it to the customer.
Note: before running the system a proper configuration of the DNS is required. The start/stop scripts make use also of ssh so a prior ssh keys exchange is required.

How to run

  1. Open a terminal inside the "run" directory
  2. Start the system with: 
    [LINUX] ./automated-car-parking-start.sh
  3. Wait until all the docker images are running...
  4. Open the ParkServiceGUI, ParkServiceStatusGUI, MockDataGUI and use them.
  5. Stop the system with: 
    [LINUX] ./automated-car-parking-stop.sh
